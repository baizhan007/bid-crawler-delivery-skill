# 交付结构规范

本规范用于最终桌面交付文件夹。重点是“每个网站独立项目”，未来流程化增加网站时，复制同一结构即可。

## 顶层结构

```text
招投标爬虫交付_项目名/
├── 交付结构说明.md
├── 网站A/
└── 网站B/
```

顶层只放总体说明和网站目录。不要把 CSV、源码、临时输出混在顶层。

## 单个网站结构

```text
网站名称/
├── 完整源码/
│   ├── 网站名称_爬虫.py
│   └── requirements.txt
├── 部署文档/
│   └── 部署文档.md
├── 配置说明/
│   └── 配置说明.md
├── 字段映射表/
│   ├── field_mapping.csv
│   └── 字段映射说明.md
├── 采集结果/
│   ├── 1、采购公告.csv
│   └── 2、结果公告.csv
└── 验收报告/
    └── acceptance_report.json
```

用户若坚持“四个字文件夹”，这里的四个必需文件夹是：

- `完整源码`
- `部署文档`
- `配置说明`
- `字段映射表`

`采集结果` 和 `验收报告` 是为了交付数据和证明验收，不替代前四个文件夹。

## 完整源码

只放本网站可独立运行的源码和依赖清单：

```text
完整源码/
├── 网站名称_爬虫.py
└── requirements.txt
```

默认不要放：

- `bid_spider/`
- `__pycache__/`
- `output_final/`
- `.idea/`

`requirements.txt` 默认只写实际需要的库，例如：

```text
requests>=2.31.0
```

只有脚本真的用到 `beautifulsoup4`、`lxml`、`pandas` 等库时才加。

## 部署文档

必须包含：

- 项目说明。
- 采集范围。
- 本次交付日期范围。
- 本次交付记录数。
- 环境要求。
- 依赖安装命令。
- 全量运行命令。
- 指定日期范围运行命令。
- 增量/断点运行命令。
- 输出文件说明。

示例：

```markdown
## 正式全量运行

```powershell
python "网站名称_爬虫.py" --days 30 --output-dir output_final
```
```

## 配置说明

必须包含：

- `BID_SPIDER_DELAY`: 请求间隔。
- `BID_SPIDER_PROXY`: 代理 IP。
- `state/crawl_state.json`: 断点状态文件。
- 栏目参数在哪里改。
- 日期范围参数怎么改。

默认不要写数据库配置，除非用户要求直接写库。

## 字段映射表

`field_mapping.csv` 至少三列：

```text
字段,源网页/接口位置,说明
```

映射说明里要写清楚：

- 标题来自哪里。
- 正文来自哪里。
- 发布时间来自哪里。
- 详情链接来自哪里。
- `html` 怎么清洗。
- `extra` 是否保存原始 JSON。

## 采集结果

默认每个网站按确认采集的采购相关栏目拆分 CSV：

```text
采集结果/1、采购公告.csv
采集结果/2、结果公告.csv
```

文件名格式为 `序号、栏目名.csv`。如果某个确认栏目本次没有数据，不生成空 CSV。

CSV 使用 `utf-8-sig` 编码，方便 Excel 打开不乱码。

## 验收报告

每个网站一个：

```text
验收报告/acceptance_report.json
```

验收报告要能回答：

- 采了多少条。
- 是否重复。
- 必填字段完整率。
- 是否在日期范围内。
- 链接是否可追溯。
- 栏目覆盖率是否达标。

## 最终清理规则

交付前删除：

- `.idea`
- `.vscode`
- `__pycache__`
- `.pytest_cache`
- `output_smoke`
- `output_test`
- 临时截图
- 旧版本源码目录
- 不在本次范围的网站目录
- 本次交付范围外的额外文件

用户给甲方看的目录应该干净到“每个文件都能解释为什么存在”。
